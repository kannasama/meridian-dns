## Workstream 2: Configuration Migration to Database

### Goal

Move runtime configuration from environment variables into the database-backed `system_config`
table. Environment variables seed initial values on first run; DB values take precedence after
that.

### Retained as Environment Variables (5)

| Variable | Required | Default |
|----------|----------|---------|
| `DNS_DB_URL` | Yes | — |
| `DNS_MASTER_KEY` / `DNS_MASTER_KEY_FILE` | Yes | — |
| `DNS_JWT_SECRET` / `DNS_JWT_SECRET_FILE` | Yes | — |
| `DNS_LOG_LEVEL` | No | `info` |
| `DNS_HTTP_PORT` | No | `8080` |

The `_FILE` variants remain supported for `DNS_MASTER_KEY` and `DNS_JWT_SECRET`.

### Migrated to Database

| Setting Key | Current Env Var | Default |
|-------------|----------------|---------|
| `http.threads` | `DNS_HTTP_THREADS` | 4 |
| `session.absolute_ttl_seconds` | `DNS_SESSION_ABSOLUTE_TTL_SECONDS` | 86400 |
| `session.cleanup_interval_seconds` | `DNS_SESSION_CLEANUP_INTERVAL_SECONDS` | 3600 |
| `apikey.cleanup_grace_seconds` | `DNS_API_KEY_CLEANUP_GRACE_SECONDS` | 300 |
| `apikey.cleanup_interval_seconds` | `DNS_API_KEY_CLEANUP_INTERVAL_SECONDS` | 3600 |
| `deployment.retention_count` | `DNS_DEPLOYMENT_RETENTION_COUNT` | 10 |
| `ui.dir` | `DNS_UI_DIR` | (empty) |
| `migrations.dir` | `DNS_MIGRATIONS_DIR` | `/opt/meridian-dns/db` |
| `sync.check_interval_seconds` | `DNS_SYNC_CHECK_INTERVAL` | 3600 |
| `audit.db_url` | `DNS_AUDIT_DB_URL` | (uses main DB) |
| `audit.stdout` | `DNS_AUDIT_STDOUT` | false |
| `audit.retention_days` | `DNS_AUDIT_RETENTION_DAYS` | 365 |
| `audit.purge_interval_seconds` | `DNS_AUDIT_PURGE_INTERVAL_SECONDS` | 86400 |

### Schema Changes

The existing `system_config` table (bootstrapped in `MigrationRunner::bootstrap()` as
`key TEXT PRIMARY KEY, value TEXT NOT NULL`) remains unchanged at bootstrap time. A versioned
migration (v007) adds optional columns:

```sql
ALTER TABLE system_config ADD COLUMN description TEXT;
ALTER TABLE system_config ADD COLUMN updated_at TIMESTAMPTZ DEFAULT now();
```

This ensures first-run setup (which uses `system_config` for `setup_completed`) is not broken —
the bootstrap creates the minimal table, setup runs against it, and the migration adds columns
afterward.

### Startup Behavior

1. Load the 5 env vars directly
2. Connect to DB
3. Run migrations (including v007 `system_config` extension)
4. For each DB-configurable setting: if no row exists in `system_config`, seed from env var
   (if set) or use compiled default
5. All subsequent reads come from `system_config` table via a `SettingsRepository`
6. `Config` object populated from DB values after seeding

### Restart-Required Settings

Some settings cannot take effect without a restart: `http.threads`, `ui.dir`, `migrations.dir`,
`audit.db_url`. These are flagged in the UI with a restart-required indicator. All others take
effect on the next maintenance cycle or immediately.

### Admin UI

Settings page (admin-only) with card-based sections grouping related settings:

- **Session & Security** — session TTL, cleanup intervals, API key grace period
- **Deployment** — retention count
- **Sync** — check interval
- **Audit** — retention days, purge interval, stdout logging
- **Paths** — UI directory, migrations directory (restart required)
- **Performance** — HTTP threads (restart required)

Each field shows current value, compiled default, and restart-required indicator where
applicable.

### ARCHITECTURE.md Updates

Document the hybrid config model: which settings are env-only, the seeding behavior, the
`system_config` table schema, and the `SettingsRepository` API.

## Workstream 3: Permissions Restructure

### Goal

Replace the fixed three-role system with discrete permissions collected into customizable named
roles, with hierarchical resource scoping at the view and zone levels.

### Permission Strings

| Category | Permissions |
|----------|------------|
| Zones | `zones.view`, `zones.create`, `zones.edit`, `zones.delete`, `zones.deploy`, `zones.rollback` |
| Records | `records.view`, `records.create`, `records.edit`, `records.delete`, `records.import` |
| Providers | `providers.view`, `providers.create`, `providers.edit`, `providers.delete` |
| Views | `views.view`, `views.create`, `views.edit`, `views.delete` |
| Variables | `variables.view`, `variables.create`, `variables.edit`, `variables.delete` |
| Git Repos | `repos.view`, `repos.create`, `repos.edit`, `repos.delete` |
| Audit | `audit.view`, `audit.export`, `audit.purge` |
| Users | `users.view`, `users.create`, `users.edit`, `users.delete` |
| Groups | `groups.view`, `groups.create`, `groups.edit`, `groups.delete` |
| Roles | `roles.view`, `roles.create`, `roles.edit`, `roles.delete` |
| Settings | `settings.view`, `settings.edit` |
| Backup | `backup.create`, `backup.restore` |

Permissions are code-defined constants, not stored in a database table.

### Default Roles

Three system roles shipped by default (non-deletable, `is_system = true`):

- **Admin** — all permissions
- **Operator** — everything except user/group/role/settings management and audit purge
- **Viewer** — all `*.view` permissions plus `audit.export`

Admins can create additional custom roles with any permission combination.

### Schema

```sql
CREATE TABLE roles (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100) UNIQUE NOT NULL,
  description TEXT,
  is_system BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE role_permissions (
  role_id INTEGER REFERENCES roles(id) ON DELETE CASCADE,
  permission VARCHAR(100) NOT NULL,
  PRIMARY KEY (role_id, permission)
);

ALTER TABLE group_members ADD COLUMN role_id INTEGER REFERENCES roles(id);
ALTER TABLE group_members ADD COLUMN scope_type VARCHAR(10);  -- NULL, 'view', or 'zone'
ALTER TABLE group_members ADD COLUMN scope_id INTEGER;        -- view or zone ID
```

### Hierarchical Scoping

Group assignments can be scoped globally (no scope), to a view, or to a specific zone:

- **Global (scope_type = NULL):** permissions apply to all resources
- **View-level (scope_type = 'view'):** permissions apply to all zones within the view
- **Zone-level (scope_type = 'zone'):** permissions apply to a specific zone only
- Zone-level takes precedence over view-level

### Resolution Logic

For a given user and resource check (e.g., "can user deploy zone X?"):

1. Collect all group memberships with their role + scope
2. Gather permissions from assignments where:
   - Scope is NULL (global), or
   - Scope matches zone X's view, or
   - Scope matches zone X directly
3. Union of all matching permissions determines access

### Migration from Current Model

The existing `role` column on `groups` maps to the three default roles. Migration:

1. Create `roles` and `role_permissions` tables
2. Seed Admin, Operator, Viewer system roles with their permissions
3. Populate `role_id` on existing `group_members` rows based on their group's current `role`
4. Drop `groups.role` column

### API Endpoints

- `GET/POST/PUT/DELETE /api/v1/roles` — role CRUD (admin-only)
- `GET /api/v1/roles/{id}/permissions` — list permissions for a role
- `PUT /api/v1/roles/{id}/permissions` — set permissions for a role
- Group assignment endpoints updated to accept `role_id`, `scope_type`, `scope_id`
- `AuthMiddleware` updated to resolve permissions instead of checking role strings

### Admin UI

- Roles management page — DataTable CRUD with permission checkbox grid
- Group assignment UI gains role dropdown + optional scope selector (view or zone picker)
- Zone switcher dropdown in top bar filtered by `zones.view` permission

## Workstream 4: OIDC & SAML Authentication

### Goal

Federated login via OIDC and SAML identity providers. Auto-provision users on first login with
IdP group to Meridian group mapping, fallback to configurable default group.

### Schema

```sql
CREATE TABLE identity_providers (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100) UNIQUE NOT NULL,
  type VARCHAR(10) NOT NULL,          -- 'oidc' or 'saml'
  is_enabled BOOLEAN DEFAULT true,
  config JSONB NOT NULL,              -- type-specific configuration
  encrypted_secret TEXT,              -- client_secret (OIDC) encrypted via CryptoService
  group_mappings JSONB,               -- IdP group → Meridian group mapping rules
  default_group_id INTEGER REFERENCES groups(id),
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

### OIDC Configuration (in `config` JSONB)

- `issuer_url` — discovery endpoint (`.well-known/openid-configuration`)
- `client_id`
- `redirect_uri` — callback URL (`/api/v1/auth/oidc/{id}/callback`)
- `scopes` — default `["openid", "profile", "email", "groups"]`
- `groups_claim` — claim path for group extraction (default: `groups`)

### SAML Configuration (in `config` JSONB)

- `entity_id` — SP entity ID
- `sso_url` — IdP SSO endpoint
- `certificate` — IdP signing certificate (PEM)
- `assertion_consumer_service_url` — callback URL
- `name_id_format`
- `group_attribute` — SAML attribute name for group extraction

### Group Mapping Rules (`group_mappings` JSONB)

```json
{
  "rules": [
    { "idp_group": "dns-admins", "meridian_group_id": 1 },
    { "idp_group": "dns-operators", "meridian_group_id": 2 },
    { "idp_group": "platform-*", "meridian_group_id": 3 }
  ]
}
```

Wildcard matching supported. If no rules match or mappings are misconfigured, user is assigned
to `default_group_id`.

### OIDC Authentication Flow

1. `GET /api/v1/auth/oidc/{id}/login` — generates state + PKCE, redirects to IdP authorize
   endpoint
2. IdP redirects back to `GET /api/v1/auth/oidc/{id}/callback` with authorization code
3. Backend exchanges code for tokens, validates ID token signature against JWKS
4. Extracts `sub`, `email`, `preferred_username`, and groups claim
5. Provisions or updates user → maps groups → issues JWT session

### SAML Authentication Flow

1. `GET /api/v1/auth/saml/{id}/login` — generates AuthnRequest, redirects to IdP SSO
2. IdP POSTs assertion to `POST /api/v1/auth/saml/{id}/acs`
3. Backend validates signature against configured certificate, checks replay cache, validates
   conditions (audience, timestamps)
4. Extracts `NameID`, attributes including group attribute
5. Provisions or updates user → maps groups → issues JWT session

### User Provisioning (First Login)

1. Look up user by `oidc_sub` or `saml_name_id`
2. If not found: create user with `auth_method` = `oidc`/`saml`, `is_active` = true, no
   password hash
3. Extract groups from claims/attributes
4. Match against mapping rules → assign to matched groups
5. If no rules match → assign to `default_group_id`
6. On subsequent logins: update email if changed, re-evaluate group mappings

### Claim/Attribute Test Diagnostic

`GET /api/v1/identity-providers/{id}/test` — initiates the auth flow with `mode=test` in state.
On callback, instead of provisioning, returns the raw decoded claims (OIDC) or attributes
(SAML) as JSON to the UI.

Admin UI displays these in a readable panel: subject, email, groups list, all other
claims/attributes. Admin uses this to build mapping rules with the exact group names the IdP
sends, eliminating guesswork.

### Login Page Changes

If any IdP is enabled, the login page shows federated login buttons below the local login form.
Each button labeled with the IdP `name` (e.g., "Sign in with Okta"). Local login remains
available unless explicitly disabled in settings.

### Admin UI

Identity Providers management page:

- DataTable CRUD for IdPs
- Provider form: type selector (OIDC/SAML), type-specific config fields, IdP group → Meridian
  group mapping rule builder, default group dropdown
- Test connection button: OIDC fetches discovery document; SAML validates certificate parsing
- Test authentication button: performs diagnostic auth flow and displays returned claims/attributes

### Dependencies

- cpp-httplib (already available) for OIDC token exchange and JWKS fetch
- OpenSSL (already linked) for SAML signature validation and OIDC JWT verification
- Existing `SamlReplayCache` used for SAML assertion replay prevention

## Workstream 5: Multi-Repo GitOps

### Goal

Git repositories as first-class entities with encrypted credentials stored in the database.
Zones reference a repo with optional branch override. Support SSH and HTTPS authentication.

### Schema

```sql
CREATE TABLE git_repos (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100) UNIQUE NOT NULL,
  remote_url TEXT NOT NULL,
  auth_type VARCHAR(10) NOT NULL,   -- 'ssh', 'https', 'none'
  encrypted_credentials TEXT,        -- AES-256-GCM via CryptoService
  default_branch VARCHAR(100) DEFAULT 'main',
  local_path TEXT,                   -- auto-generated if NULL
  known_hosts TEXT,                  -- SSH host verification
  is_enabled BOOLEAN DEFAULT true,
  last_sync_at TIMESTAMPTZ,
  last_sync_status VARCHAR(20),     -- 'success', 'failed'
  last_sync_error TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

ALTER TABLE zones ADD COLUMN git_repo_id INTEGER REFERENCES git_repos(id);
ALTER TABLE zones ADD COLUMN git_branch VARCHAR(100);  -- NULL = use repo default
```

### Credential Storage

Credentials encrypted via `CryptoService` (AES-256-GCM) before storage. Decrypted JSONB:

**SSH:**
```json
{
  "private_key": "-----BEGIN OPENSSH PRIVATE KEY-----\n...",
  "passphrase": ""
}
```

**HTTPS (personal access token):**
```json
{
  "username": "token-user",
  "token": "ghp_xxxxxxxxxxxx"
}
```

### GitOpsMirror Refactoring

Current `GitOpsMirror` is a singleton managing one repo. Refactored to:

- **`GitRepoManager`** — manages multiple `GitRepoMirror` instances, one per enabled
  `git_repos` row
- **`GitRepoMirror`** — wraps a single repo (clone/open, pull, commit, push). Essentially the
  current `GitOpsMirror` logic extracted per-instance
- On startup, `GitRepoManager` loads all enabled repos from DB, initializes each mirror
- Hot-reload: when an admin adds/edits/removes a repo via UI, `GitRepoManager` re-initializes
  the affected mirror without restarting

### Local Path Management

Each repo gets an auto-generated local path under a configurable base directory (DB setting,
default `/var/meridian-dns/repos/`). Path derived from repo ID: `/var/meridian-dns/repos/{id}/`.
Admin can override per-repo.

### Zone Snapshot Path

Same structure as current: `{view_name}/{zone_name}.json`. Branch determined by zone's
`git_branch` override or repo's `default_branch`.

### Branch Handling

- Repo has a `default_branch` (e.g., `main`)
- Zone can override with `git_branch` (e.g., `production`, `staging`)
- Multiple zones in the same repo on different branches: `GitRepoMirror` handles branch
  switching per-commit operation (checkout → write → commit → push → checkout default).
  Operations are mutex-serialized per repo, so branch switching is safe.
- If branch switching overhead becomes a concern, the manager could maintain separate worktrees
  per branch. Start simple with the checkout-based approach.

### HTTPS Authentication

Adds `git_credential_userpass_plaintext_new` in the libgit2 credentials callback for HTTPS
remotes, using the decrypted token from DB.

### Migration from Environment Variables

If `DNS_GIT_REMOTE_URL` is set on first run after upgrade:

1. Create a `git_repos` row with the env var values (remote URL, SSH key path contents,
   known hosts)
2. SSH key file contents read and encrypted into DB
3. Assign all existing zones to this repo
4. Log a deprecation notice — env vars for GitOps ignored on subsequent starts

### API Endpoints

- `GET/POST/PUT/DELETE /api/v1/git-repos` — repo CRUD
- `POST /api/v1/git-repos/{id}/test` — test connection (clone to temp dir, verify auth)
- `POST /api/v1/git-repos/{id}/sync` — manual pull

### Admin UI

Git Repositories management page — DataTable CRUD with drawer form. Fields: name, remote URL,
auth type (SSH/HTTPS/None), credentials (key textarea or token input), default branch, known
hosts. Status indicator showing last sync result. Test connection button in the form.

Zone form updated with git repo dropdown and optional branch override field.

## Workstream 6: Config Backup & Restore

### Goal

Full system export excluding encrypted credentials. Two restore paths: GitOps-based and file
upload. On restore, admin re-enters all credentials.

### Export Format

Single JSON file with versioned structure:

```json
{
  "version": 1,
  "exported_at": "2026-03-08T12:00:00Z",
  "exported_by": "admin",
  "meridian_version": "1.0.0",
  "settings": {},
  "roles": [],
  "groups": [],
  "users": [],
  "identity_providers": [],
  "git_repos": [],
  "providers": [],
  "views": [],
  "zones": [],
  "records": [],
  "variables": []
}
```

### Included vs Excluded Per Entity

| Entity | Included | Excluded |
|--------|----------|----------|
| Settings | All key/value pairs | — |
| Roles | Name, permissions, is_system flag | Internal IDs (re-mapped) |
| Groups | Name, description, role assignments | — |
| Users | Username, email, auth_method, is_active | Password hashes, oidc_sub, saml_name_id |
| Identity Providers | Name, type, config (non-secret) | encrypted_secret, client_secret |
| Git Repos | Name, remote_url, auth_type, default_branch | encrypted_credentials, known_hosts |
| Providers | Name, type, endpoint | encrypted_config (tokens) |
| Views | Name, description, provider assignments | — |
| Zones | Name, view (by name), git repo (by name), branch, flags | sync_status, sync_checked_at |
| Records | All fields by zone name | provider_meta (provider-specific) |
| Variables | Name, value, scope, zone (by name) | — |

References use **names, not IDs** for portability across instances. No audit log, deployment
history, sessions, or API keys — these are operational/ephemeral data.

### Export Endpoint

`GET /api/v1/backup/export` — requires `backup.create` permission. Returns JSON file with
`Content-Disposition: attachment; filename="meridian-backup-{timestamp}.json"`.

### GitOps Backup

The export can optionally be committed to a designated git repo. A `system_config` setting
(`backup.git_repo_id`) identifies which repo stores config backups. The JSON is written to a
configurable path (default: `_system/config-backup.json`) and committed with a timestamped
message.

Automatic scheduled backup: maintenance task periodically exports and commits (interval
configurable in settings, default disabled). Manual trigger via UI button.

### Restore — File Upload

`POST /api/v1/backup/restore` — multipart file upload, requires `backup.restore` permission.

- **Preview mode (default):** parses file and returns summary of what will be
  created/updated/skipped without applying changes
- **Apply mode (`?apply=true`):** executes the restore within a transaction

### Restore — GitOps Pull

`POST /api/v1/backup/restore-from-repo` — pulls latest backup from the configured backup repo.
Same preview/apply flow as file upload.

### Restore Behavior

1. Parse and validate the backup file (`version` field compatibility check)
2. For each entity type, in dependency order: settings → roles → groups → users → providers →
   git repos → views → zones → records → variables
3. Match by name to existing entities:
   - **Exists:** update non-credential fields
   - **Doesn't exist:** create with placeholder/empty credentials
4. Flag entities needing credential re-entry (providers, git repos, IdPs) — returned in the
   restore response and shown in the UI
5. Users restored without password hashes — admin must trigger password resets or users
   re-authenticate via IdP
6. Entire restore is transactional — all or nothing

### Zone-Level Export/Import

In addition to full system backup:

- `GET /api/v1/zones/{id}/export` — exports a single zone with its records and variables
- `POST /api/v1/zones/{id}/import` — restores a zone from an exported JSON

Covers moving or duplicating a zone without a full system restore.

### Documentation

Clearly documented that restoration requires new encryption keys and re-entry of all
credentials. The export file contains no encrypted data and is safe to store without key
management concerns.

### Admin UI

Backup & Restore page (admin-only, under settings):

- Export button with optional "commit to GitOps" checkbox
- Upload area for restore from file
- Restore from repo button (if backup repo configured)
- Preview table showing what will change before applying
- Post-restore: list of entities needing credential re-entry, with links to their edit forms

## Workstream 7: Zone Initial Snapshot

### Goal

Ensure every zone has at least one GitOps snapshot for rollback and backup purposes, even if
the zone was never deployed through Meridian.

### Auto-Capture (One-Time)

During the background sync check (existing `MaintenanceScheduler` task), after a zone's sync
status is confirmed as `in_sync`:

1. Check if the zone has a `git_repo_id` assigned — skip if no repo configured
2. Query `deployments` table for the zone — if any deployment snapshot exists, skip
3. Pull current records from the provider via `IProvider::listRecords()`
4. Build the snapshot JSON in the same format as a normal deployment snapshot (zone name, view,
   records with expanded values, timestamp, `generated_by: "auto-capture"`)
5. Commit to the zone's configured git repo and branch
6. Create a deployment record with `seq = 0` and a note indicating auto-capture

**Why seq 0:** Normal deployments start at seq 1 (auto-increment). Using 0 clearly
distinguishes auto-captured baselines from intentional deployments.

Auto-capture fires **once only** — when no prior snapshot exists and sync confirms in-sync.
Subsequent snapshots only come from Meridian deployments or manual capture.

### Manual Capture

"Capture Current State" button on ZoneDetailView, visible when the zone has a git repo
assigned.

`POST /api/v1/zones/{id}/capture` — requires `zones.deploy` permission.

1. Pull current records from the provider
2. Build and commit snapshot to GitOps
3. Create a deployment record (next seq number) with note `"manual-capture"`
4. Return the snapshot summary

### Differences from Normal Deployment

- No diff computation or push to provider — the provider already has this state
- No variable expansion step — records captured as-is from the provider
- Deployment record flagged as a capture, not a push

### Audit Trail

Both auto-capture and manual capture generate audit log entries with
`operation: "zone.capture"` and `entity_type: "zone"`, recording the source (`auto` or
`manual`) and the number of records captured.

### UI Changes

- ZoneDetailView: "Capture Current State" button in page header actions, alongside Deploy link
- Deployment history: auto-captured entries show a distinct badge ("Baseline" or "Captured")
- Deployment detail: shows `generated_by` to distinguish auto-capture, manual capture, and
  regular deployments

### Health & Readiness Probes

Two new unauthenticated endpoints alongside the existing `GET /api/v1/health` (kept for
backward compatibility).

**`GET /api/v1/health/live` — liveness probe (no auth):**

- Returns `200 OK` with `{"status": "alive", "version": "1.0.0"}` if the HTTP server is
  responsive
- No DB check, no external dependency checks — purely "is the process alive and serving"
- Used by container orchestrators to decide whether to restart the container

**`GET /api/v1/health/ready` — readiness probe (no auth):**

- Checks DB connection pool (execute `SELECT 1`)
- Checks enabled git repos sync status (how many healthy vs failed)
- Checks enabled providers reachability (how many healthy vs degraded)
- Returns `200` with component status JSON if DB is reachable, `503` if DB is down

```json
{
  "status": "healthy",
  "version": "1.0.0",
  "components": {
    "database": "healthy",
    "git_repos": { "healthy": 2, "failed": 0 },
    "providers": { "healthy": 3, "degraded": 1 }
  }
}
```

- `status` is `"healthy"` when DB is up and all components functional, `"degraded"` when DB is
  up but some repos/providers fail, `"unhealthy"` when DB is unreachable
- The readiness endpoint is informational for observability dashboards — upstream provider
  issues should not impact application availability or load balancing decisions

Existing `/api/v1/health` updated to include `"version"` field but otherwise unchanged.

**Dockerfile HEALTHCHECK** updated to use `/api/v1/health/live` — readiness failures from
upstream providers should not cause container restarts.

### Security Headers

Two-layer approach: basic headers enforced by the application, full CSP delegated to the
reverse proxy.

**In-app headers (Crow middleware):**

A `SecurityHeadersMiddleware` struct added to Crow's middleware chain, injecting on every
response:

- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `X-XSS-Protection: 0` (disabled — CSP supersedes this; the `1` mode can introduce
  vulnerabilities)
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy: camera=(), microphone=(), geolocation=()`

These are safe defaults that protect against clickjacking and MIME sniffing without breaking
any functionality.

**Reverse proxy headers (documented in DEPLOYMENT.md):**

The nginx and Traefik examples include recommended additions:

- `Content-Security-Policy` — tailored for PrimeVue (allows inline styles from the theme
  system, restricts scripts to `'self'`)
- `Strict-Transport-Security` — HSTS with a reasonable max-age
- `X-Robots-Tag: noindex, nofollow` — DNS management UIs should not be indexed

Documentation explains why CSP lives in the reverse proxy rather than the app (CSP is
deployment-specific — allowed origins depend on where the instance is hosted).
