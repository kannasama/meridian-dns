## Overview

v1.1 introduces four feature groups that extend Meridian DNS with operational quality-of-life
improvements and an architectural foundation for community-contributed providers. All groups are
independently releasable and do not block each other.

| # | Group | Summary |
|---|-------|---------|
| 1 | Templates & Snippets | Reusable record sets and zone blueprints with advisory compliance checks |
| 2 | Dynamic Variables | Built-in `{{sys.*}}` variables and user-defined dynamic variable types |
| 3 | QoL Features | Clone zone, global search, tags, snippets, BIND export, validation, bulk TTL |
| 4 | Pluggable Providers | Generic REST adapter + subprocess adapter with definition management UI |

## Architectural Constraints

**Database as source of truth.** All runtime state — including provider definitions, template
definitions, and tag vocabulary — lives in PostgreSQL. Local file dependencies (beyond container
image contents) are avoided to support multi-instance deployments where all instances share
a single database backend.

The one accepted exception: subprocess provider binaries must be present in the container image.
All instances run the same image, so binary availability is guaranteed without coordination.

**`{{sys.serial}}` counter persistence.** The SOA serial counter (`YYYYMMDD` + two-digit suffix)
must be durable across instances. It is stored in the `system_config` table as two keys:
`serial_counter_date` (the last date a serial was issued, UTC) and `serial_counter_seq` (the
current two-digit suffix for that date). Both are updated atomically within the same transaction
as the deployment that consumes the serial. If `serial_counter_date` differs from today's UTC
date, the seq resets to `00`. This prevents duplicate serials from concurrent deployments on
the same calendar day.

## Group 1: Templates & Snippets

### Concepts

**Snippets** are named, reusable record sets. A snippet contains an ordered list of record
templates (name, type, TTL, value_template, priority) identical in structure to zone records.
Applying a snippet to a zone bulk-inserts those records as a one-shot operation — no ongoing
relationship to the source snippet.

**SOA Presets** are named SOA timing configurations: `mname`, `rname`, `refresh`, `retry`,
`expire`, `minimum`, `default_ttl`. Both `mname` and `rname` support `{{var}}` and `{{sys.*}}`
placeholders, allowing a single preset to serve multiple zones. A preset is selectable per zone
and takes effect when `manage_soa = true`.

**Zone Templates** combine an optional SOA preset with an ordered list of snippets. Applying a
template to a zone instantiates all snippet records into the zone. Templates are the primary
mechanism for enforcing a standard starting configuration across many zones.

### Zone–Template Relationship

When a zone is created from a template (or a template is pushed to a zone), `zones.template_id`
is set. This link is the basis for compliance checks.

The relationship is **advisory, not automatic**. Template modifications do not cascade to zones.
Instead, zones linked to a modified template are flagged (`zones.template_check_pending = true`)
for operator review.

### Compliance Check

A compliance check diffs a zone's current records against the template's current expected record
set. The result surfaces **Add** and **Update** actions only — extra records in the zone beyond
the template are not considered violations and are not flagged for deletion.

The compliance check endpoint (`GET /api/v1/zones/{id}/template/check`) is callable **on
demand at any time**, not only when `template_check_pending = true`. Calling it does not
automatically clear the flag — only the Apply, Ignore, or Unlink actions clear
`template_check_pending`. This allows operators to re-check after partial application without
losing the pending state.

The compliance check response reuses the existing `PreviewResult` JSON structure (the same
shape returned by `POST /zones/{id}/preview`) restricted to **Add** and **Update** actions
only. This allows the compliance diff UI to reuse the same colour-coded diff table as the
deployment preview with no new client-side rendering logic.

The operator reviews the compliance diff and chooses one of:

| Action | Effect |
|--------|--------|
| **Apply** (selected records) | Creates or updates the chosen records in the zone. Request body: `{ "records": [{ "name": "string", "type": "string" }] }` — identifies records by name+type from the diff result. |
| **Ignore** | Dismisses this check cycle; `template_check_pending` cleared; zone stays linked; re-flagged on next template modification |
| **Unlink** | Clears `template_check_pending`; sets `zones.template_id = NULL`; zone excluded from all future compliance cycles |

### Push Template Action

"Push template" is the universal entry point for linking or relinking a zone to a template. It
applies to any zone — newly created, previously unlinked, or switching to a different template.

**Linked push (default):** Sets `zones.template_id` (replacing any existing link if the zone
was previously linked to a different template), runs an immediate compliance check for operator
review, and enrolls the zone in future compliance cycles.

**Unlinked push (opt-in flag):** Applies the template's records as a one-shot action without
setting `zones.template_id`. The zone's existing `template_id` is not modified. Equivalent to
applying all of the template's snippets in sequence. No future compliance relationship is
created.

### UI

- **Snippets** — Admin → Snippets: list, create, edit, delete. Each snippet has a record editor
  identical to the zone record editor. An "Insert snippet" button on the zone detail Records tab
  opens a picker.
- **SOA Presets** — Admin → SOA Presets: list, create, edit, delete. Selectable on the zone
  form when `manage_soa = true`.
- **Zone Templates** — Admin → Templates: list, create, edit, delete. Template editor shows an
  ordered snippet list (drag-to-reorder) and an optional SOA preset selector.
- **Compliance check** — flagged zones display a banner on the zone detail page. The compliance
  diff uses the same colour-coded Add/Update table as the deployment preview.

## Group 2: Dynamic Variables

### Resolver Chain

`VariableEngine::expand()` resolves `{{var}}` placeholders in this order:

1. **Zone-scoped static variables** — highest precedence
2. **Global static variables**
3. **Built-in system variables** (`{{sys.*}}`) — resolved at deploy time, no DB lookup
4. **User-defined dynamic variables** — resolved at deploy time using a format expression

An unresolved placeholder at any tier throws `UnresolvedVariableError` as before.

**Regex change required.** The existing tokeniser regex `[A-Za-z0-9_]+` does not match the dot
in `sys.date`, `sys.serial`, etc. The regex must be extended to `[A-Za-z0-9_.]+` to handle
namespaced variable names. This is a one-line change to `VariableEngine.cpp`. User-defined
variable names (letters, digits, underscores) are unaffected — the dot remains reserved for
built-in system namespacing.

### Built-in System Variables

| Variable | Resolved value | Example |
|----------|---------------|---------|
| `{{sys.date}}` | `YYYYMMDD` | `20260318` |
| `{{sys.datetime}}` | `YYYYMMDDHHmmss` | `20260318143022` |
| `{{sys.serial}}` | `YYYYMMDD00`, auto-incremented within the same calendar day | `2026031800` |
| `{{sys.timestamp}}` | Unix epoch seconds | `1742308222` |
| `{{sys.zone}}` | The zone name being deployed | `example.com` |

`{{sys.serial}}` increments the two-digit suffix (00–99) when multiple deployments occur on the
same calendar day. The counter resets at midnight UTC. See Architectural Constraints for the
durable storage mechanism.

### User-Defined Dynamic Variables

Stored in the `variables` table with `variable_kind = 'dynamic'` and a `dynamic_format TEXT`
field containing a `strftime`-style format string. Resolved at deploy time against UTC.

The existing `type` column (ENUM: `ipv4`, `ipv6`, `target`, `string`) is set to `'string'` for
all dynamic variables. The `type` field has no semantic meaning for dynamic variables — it
satisfies the NOT NULL constraint and is ignored by the resolver.

The v017 migration adds a CHECK constraint: `CHECK (variable_kind = 'static' OR dynamic_format IS NOT NULL)`.
This prevents a `dynamic` variable from being saved without a format string, which would cause
a runtime panic during deployment.

Example: `dynamic_format = '%Y%m%d%H%M%S'` → resolves to `20260318143022`.

### UI

The Variables view gains two tabs:

**Static** — existing behaviour: global and zone-scoped, editable, `{{var}}` autocomplete.

**Dynamic** — two sub-groups:

- **Built-in (`{{sys.*}}`)** — non-editable rows; columns: name, description, format, and a
  **Preview** column showing the current resolved value evaluated at page load.
- **User-defined** — editable rows with the same columns + Preview. The Preview evaluates the
  `dynamic_format` on load so users can validate their format string before use.

Both sub-groups are visible in the variable browser/autocomplete with a visual indicator
distinguishing dynamic from static variables.

## Group 3: QoL Features

### Clone Zone

`POST /zones/{id}/clone` copies the source zone's records (unexpanded templates, raw
`value_template` strings) into a new zone. The user provides a new zone name and view
assignment. No link to the source zone after creation. SOA preset and template assignment are
not copied — the new zone starts with `template_id = NULL`.

The cloned zone starts with `template_id = NULL` and `soa_preset_id = NULL` — neither the
template link nor the SOA preset is inherited from the source zone.

UI: "Clone zone" action on the zone detail page or zone list row menu.

### Global Record Search

`GET /api/v1/search/records?q=&type=&zone_id=&view_id=` searches `records.name`,
`records.value_template`, and `zones.name` with optional type/zone/view filters. Returns
records with parent zone context (zone name, view name).

UI: top-bar search field or dedicated Search view. Results link directly to the zone detail
page at the relevant record.

### Zone Tags

`zones.tags TEXT[]` column, GIN-indexed. Tags are free-form strings requiring no upfront
creation. When a user enters a tag, it is added to the zone and the `tags` vocabulary table is
updated automatically (upsert on `tag_name`).

**Tag vocabulary table** (`tags`): `(id, name, created_at)`. Maintains the distinct tag set
for autocomplete and management purposes.

**Tag management** — Admin → Tags: list all tags with per-tag zone count, rename, delete.
Deleting a tag from the vocabulary performs a multi-row `UPDATE zones SET tags = array_remove(tags, $tag)`
across all zones — there is no foreign key cascade, as `zones.tags` is a `TEXT[]` column.
Provides housekeeping without requiring tags to be pre-created.

UI: tag chips on zone list and zone detail. Zone list is filterable by tag. Tag input uses
autocomplete against the vocabulary.

### Record Snippets

Described in Group 1. Accessible from any zone detail view via "Insert snippet" on the Records
tab. Snippets are one-shot: applying a snippet does not create an ongoing link.

### Zone Export (BIND Format)

`GET /api/v1/zones/{id}/export?format=bind` expands all record templates using current variable
values and serialises the result as a standard BIND zone file. Returns `text/plain` in the
response body with a `Content-Disposition: attachment` header.

**Use case:** Meridian DNS as source of truth while serving zones via a self-managed BIND
instance. Fully automatable via API key authentication — `curl` the endpoint, write the file,
reload `named`. Requires at minimum the `viewer` role (same as all read-only endpoints).

Note: webhook/script trigger support for automated BIND reload is deferred to v1.2. See
[v1.2 Backlog](#v12-backlog).

### Record Validation Warnings

Applied server-side at record save time. Validation results are returned in the API response
alongside the saved record.

| Check | Severity | Description |
|-------|----------|-------------|
| CNAME coexistence | **Error** (blocking) | CNAME at a name that already has any other record type, or vice versa (RFC 1034) |
| Multiple CNAMEs | **Error** (blocking) | More than one CNAME at the same name |
| Missing trailing dot | **Warning** | Values that appear to be FQDNs (contain a dot, do not end with `.`) in record types where FQDNs are expected (CNAME, MX, NS, SRV target). Not applied to A, AAAA, or TXT. |
| Multiple SOA records | **Warning** | More than one SOA record in a zone (relevant when `manage_soa = true`) |

Round-robin and stacked-priority records (A, AAAA, MX, TXT, etc.) are explicitly not flagged
as duplicates — multiple records of the same type at the same name are valid and common.

### Bulk TTL Adjustment

`POST /zones/{id}/records/bulk-ttl` with body `{ "ttl": N, "filter_type": "A" }` (type filter
optional). Updates TTL on all matching records in the zone in a single operation.

UI: "Bulk adjust TTL" button on the zone detail Records tab. Opens a modal with a TTL input
and an optional record-type filter dropdown.

## Group 4: Pluggable Provider System

### Overview

Two new `IProvider` implementations extend the provider system to support community-contributed
and custom providers without requiring a full application rebuild:

- **`GenericRestProvider`** — driven by a JSON definition describing a REST DNS API
- **`SubprocessProvider`** — delegates to an external binary via a JSON-RPC protocol over stdin/stdout

Both appear identically to compiled providers in the health dashboard and deployment pipeline.

### Provider Definitions

A provider definition is a JSON document that describes how to interact with a DNS provider's
API. Definitions are distributed as `.json` files (via GitHub, a future community registry, or
direct authorship) and imported into Meridian via the UI or API.

**`provider_definitions` table:**

```sql
CREATE TABLE provider_definitions (
  id          BIGSERIAL PRIMARY KEY,
  name        TEXT NOT NULL,
  type_slug   TEXT NOT NULL UNIQUE,   -- stable identifier, e.g. "route53"
  version     TEXT NOT NULL,
  definition  JSONB NOT NULL,
  source_url  TEXT,
  imported_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

The `providers` table gains `definition_id BIGINT REFERENCES provider_definitions(id)`.
Compiled providers (PowerDNS, Cloudflare, DigitalOcean) have `definition_id = NULL`.

The `provider_type` ENUM is extended in v018 to add `'generic_rest'` and `'subprocess'`
values. `ProviderFactory` uses `providers.type` to select the `IProvider` implementation:
`'generic_rest'` → `GenericRestProvider`, `'subprocess'` → `SubprocessProvider`. Compiled
provider type values are unchanged.

**ProviderFactory definition passthrough.** For `generic_rest` and `subprocess` provider types,
the caller (ApiServer or DeploymentEngine) is responsible for pre-fetching
`provider_definitions.definition` from the DB and passing it to `ProviderFactory::create()`
via the existing `jConfig` parameter. The factory signature does not change. This keeps the
factory stateless and avoids injecting a DB dependency into it.

Uploading a definition with an existing `type_slug` offers an in-place update rather than
creating a duplicate row.

### Generic REST Adapter

`GenericRestProvider` reads its behaviour from `providers.definition_id → provider_definitions.definition`.

The definition JSONB schema covers:

- **Authentication** — `api_key_header`, `bearer_token`, or `basic_auth`
- **Endpoints** — URL patterns for list zones, list records, create/update/delete record;
  parameterised with `{zone_id}`, `{record_id}`, `{zone_name}`
- **Response mapping** — JSONPath expressions to extract record fields (name, type, TTL, value,
  priority, record ID) from API responses
- **Pagination** — optional cursor or offset/limit strategy
- **Zone ID resolution** — how to map a zone name to the provider's internal identifier

### Subprocess Adapter

`SubprocessProvider` spawns a provider binary and communicates over stdin/stdout using a
line-delimited JSON-RPC protocol. The binary path is specified in the definition and is
relative to the container image filesystem.

The protocol defines a fixed method set mirroring `IProvider`:

| Method | Description |
|--------|-------------|
| `listRecords` | Returns all records for a zone |
| `createRecord` | Creates a record; returns provider record ID |
| `updateRecord` | Updates an existing record by provider record ID |
| `deleteRecord` | Deletes a record by provider record ID |
| `testConnectivity` | Health check; returns `ok` or error detail |

**Multi-instance note:** Subprocess provider binaries must be present in the container image.
This is the one accepted exception to the database-as-source-of-truth constraint — binary
availability is guaranteed because all instances run the same image.

### Definition Management UI

Admin → Provider Definitions:

| Action | Description |
|--------|-------------|
| **List** | All installed definitions with name, type slug, version, import date, source URL, and active instance count |
| **Edit** | In-browser JSON editor (Monaco or CodeMirror). Saves in place — no new row created. Active instances pick up the updated definition on their next operation. |
| **Export** | Downloads the definition as a `.json` file for sharing, version control, or community submission |
| **Delete** | Blocked if any provider instances reference the definition (error lists dependent instances). Allowed once all instances are removed. |
| **Upload** | `POST /api/v1/provider-definitions`. If `type_slug` already exists, offers in-place update. |

The edit → export workflow supports iterative definition development without database clutter:
upload once, refine in the editor, test against a real provider, export the finished definition.

## Schema Changes Summary

Migrations are applied in strict numeric sequence. v016 must be applied before v017, which must
be applied before v018. No gaps exist between v015 (last v1.0 migration) and v016.

| Migration | Table | Change |
|-----------|-------|--------|
| v016 | `soa_presets` | New table: id, name, mname_template, rname_template, refresh, retry, expire, minimum, default_ttl |
| v016 | `snippets` | New table: id, name, description |
| v016 | `snippet_records` | New table: id, snippet_id FK, name, type, ttl, value_template, priority, sort_order INTEGER NOT NULL. `provider_meta` is intentionally excluded — snippets define DNS structure, not provider-specific metadata. |
| v016 | `zone_templates` | New table: id, name, description, soa_preset_id FK nullable |
| v016 | `zone_template_snippets` | New join table: template_id FK, snippet_id FK, sort_order INTEGER NOT NULL |
| v016 | `zones` | Add `template_id BIGINT FK nullable`, `template_check_pending BOOLEAN DEFAULT FALSE`, `tags TEXT[]` GIN-indexed |
| v017 | `variable_kind` | `CREATE TYPE variable_kind AS ENUM ('static', 'dynamic')` |
| v017 | `variables` | Add `variable_kind variable_kind NOT NULL DEFAULT 'static'`, `dynamic_format TEXT`, `CHECK (variable_kind = 'static' OR dynamic_format IS NOT NULL)` |
| v017 | `tags` | New table: id, name UNIQUE, created_at |
| v018 | `provider_type` | Extend ENUM: add `'generic_rest'`, `'subprocess'` |
| v018 | `provider_definitions` | New table (see Group 4) |
| v018 | `providers` | Add `definition_id BIGINT REFERENCES provider_definitions(id) ON DELETE RESTRICT` (nullable) |

## v1.2 Backlog

- **BIND export webhook/script trigger** — post-deployment trigger (webhook URL or local script
  path) fired after a successful zone export or push, enabling automated BIND reload workflows
  (e.g., `scp` zone file to BIND server, call `rndc reload`). Requires a trigger configuration
  per zone or globally. Deferred because it introduces external network dependencies and
  requires careful security scoping.
- **Community provider registry** — a hosted index of community-contributed provider definitions
  browsable and installable directly from the Admin → Provider Definitions UI, without manual
  JSON file downloads.
- **Approval workflow** — operator generates a deployment preview, saves it for admin review,
  admin approves or rejects before the push is allowed. Requires a `pending_deployments` table
  and notification mechanism.
