## 1. Bug Fixes

### 1.1 Login: Immediate Logout with "Session Expired" + Error Clears Too Quickly

**Problem:** The `apiRequest` 401 handler in `ui/src/api/client.ts` (lines 44-48) applies
globally to ALL 401 responses, including from the login endpoint itself:

```typescript
if (response.status === 401) {
    localStorage.removeItem('jwt')
    window.location.href = '/login'
    throw new ApiRequestError(401, { error: 'unauthorized', message: 'Session expired' })
}
```

This causes three cascading problems:

1. **Login failures show wrong message.** When `POST /auth/local/login` returns 401 for
   invalid credentials, the handler discards the actual response body (`"Invalid username or
   password"`) and replaces it with a hardcoded `"Session expired"`. The user sees "Session
   expired" instead of the real error.

2. **Error clears immediately.** `window.location.href = '/login'` triggers a full page
   reload, wiping all Vue state. The error message set in `LoginView.vue`'s `errorMessage`
   ref is destroyed before the user can read it.

3. **Stale token cascade.** If a stale JWT exists in localStorage from a previous session
   (server restart, expired token), the router guard's `auth.hydrate()` call triggers
   `GET /me` → 401 → hard redirect. This is fine for initial load, but the hard redirect
   approach is heavy-handed and causes flash-of-content.

**Fix (`ui/src/api/client.ts`):**

```typescript
if (response.status === 401) {
  const body = await response.json().catch(() => ({
    error: 'unauthorized',
    message: 'Session expired',
  }))

  // Only redirect for session expiry (non-auth endpoints where we had a token)
  const hadToken = !!localStorage.getItem('jwt')
  localStorage.removeItem('jwt')

  if (hadToken && !path.startsWith('/auth/')) {
    window.location.href = '/login'
  }

  throw new ApiRequestError(401, body)
}
```

Key changes:
- **Read actual response body** before throwing, so login errors show the real message.
- **Only hard-redirect for session expiry** — when the request was to a non-auth endpoint
  and there was a JWT stored (meaning the user was logged in and the session expired).
- **Do not redirect for auth endpoint 401s** — login failures, password change failures,
  etc. stay on the current page with the error message intact.

This means login errors now persist in the UI until the user's next submit attempt (which
clears them at the top of `handleLogin()`). No page reload, no message loss.

### 1.2 Setup Redirect Loop After First-Run User Creation

**Problem:** After completing initial setup, the app fails to redirect to the dashboard.
The module-level `setupRequired` variable in `router/index.ts` is set to `true` on first
navigation guard check and never updated after setup completes. When `router.push('/')`
fires after setup, the guard sees `setupRequired` is still `true` and redirects back to
`/setup`.

**Fix:** Export a function from the router module to reset setup state, and call it from
`SetupView.vue` after successful setup:

In `router/index.ts`:

```typescript
export function markSetupComplete() {
  setupRequired = false
}
```

In `SetupView.vue`:

```typescript
import { markSetupComplete } from '../router'

// After successful setup:
await completeSetup({ ... })
markSetupComplete()
await auth.login(username.value, password.value)
router.push('/')
```

### 1.3 Sync Status "Last Checked" Shows "20499d ago"

**Problem:** The backend returns `sync_checked_at` as Unix epoch seconds (a number, e.g.
`1741363200`). The frontend's `relativeTime()` function passes this to `new Date(iso)` which
interprets it as ISO string parsing. JavaScript's `new Date(1741363200)` treats the number as
milliseconds since epoch (→ January 1970), producing a ~20,000-day offset.

Additionally, the bulk sync-check endpoint (`POST /api/v1/zones/sync-check`) omits
`sync_checked_at` from its response entirely, so `refreshAllZones()` writes `undefined` back
to the zone objects.

**Fix (backend — two changes):**

1. In `ZoneRoutes.cpp` bulk sync-check handler (line 70), after `updateSyncStatus`, re-fetch
   the zone to include `sync_checked_at` in the response:

   ```cpp
   auto oUpdated = _zrRepo.findById(zone.iId);
   nlohmann::json jEntry = {{"zone_id", zone.iId}, {"sync_status", sStatus}};
   if (oUpdated && oUpdated->oSyncCheckedAt.has_value()) {
     jEntry["sync_checked_at"] = std::chrono::duration_cast<std::chrono::seconds>(
         oUpdated->oSyncCheckedAt->time_since_epoch()).count();
   } else {
     jEntry["sync_checked_at"] = nullptr;
   }
   jResults.push_back(jEntry);
   ```

2. No change needed to the epoch format itself — fix on frontend.

**Fix (frontend — `DashboardView.vue`):**

Change `relativeTime()` to handle both numeric epoch seconds and ISO strings:

```typescript
function relativeTime(ts: string | number | null | undefined): string {
  if (ts == null) return 'Never'
  const ms = typeof ts === 'number' ? ts * 1000 : new Date(ts).getTime()
  const diff = Date.now() - ms
  const mins = Math.floor(diff / 60000)
  if (mins < 1) return 'Just now'
  if (mins < 60) return `${mins}m ago`
  const hours = Math.floor(mins / 60)
  if (hours < 24) return `${hours}h ago`
  return `${Math.floor(hours / 24)}d ago`
}
```

### 1.4 Cloudflare Deployment Does Not Delete Records

**Problem:** When drift records are marked for deletion during deployment push, the
`DeploymentEngine` constructs a composite record ID (`name/type/value`) at line 220. This
format works for PowerDNS (which uses composite keys in its API) but fails for Cloudflare
(which requires opaque UUID record IDs from its API) and DigitalOcean (which uses numeric IDs).

```cpp
// Current — broken for Cloudflare/DigitalOcean
std::string sRecordId = diff.sName + "/" + diff.sType + "/" + diff.sProviderValue;
```

**Root cause:** `RecordDiff` does not carry the provider's native record ID. The live record
data (which has `sProviderRecordId` on `DnsRecord`) is used to build the diff, but the ID is
discarded during diff computation.

**Fix (three changes):**

1. **Add `sProviderRecordId` to `RecordDiff`** in `include/common/Types.hpp`:

   ```cpp
   struct RecordDiff {
     DiffAction action;
     std::string sName;
     std::string sType;
     std::string sProviderValue;
     std::string sSourceValue;
     uint32_t uTtl = 300;
     int iPriority = 0;
     nlohmann::json jProviderMeta;
     std::string sProviderRecordId;  // Provider's native record ID for deletes
   };
   ```

2. **Propagate provider record ID in `DiffEngine::computeDiff()`** — when creating Drift and
   Update diffs from live records, copy `dr.sProviderRecordId` into the `RecordDiff`:

   ```cpp
   // In the drift loop (line 143-151):
   rd.sProviderRecordId = dr.sProviderRecordId;

   // In the update pairing (line 93-102), look up the live record to get its ID:
   // Find the matching live record for the provider value
   for (const auto& liveDr : vLive) {
     if (liveDr.sName == dr.sName && liveDr.sType == dr.sType
         && liveDr.sValue == sProviderValue) {
       rd.sProviderRecordId = liveDr.sProviderRecordId;
       break;
     }
   }
   ```

3. **Use `sProviderRecordId` in `DeploymentEngine::push()`** — replace the composite ID
   construction with the provider's native ID:

   ```cpp
   } else if (itAction->second == "delete") {
     bool bDeleted = upProvider->deleteRecord(oZone->sName, diff.sProviderRecordId);
     // ...
   }
   ```

   For PowerDNS, `sProviderRecordId` is already set to the composite format by
   `PowerDnsProvider::listRecords()`. For Cloudflare and DigitalOcean, it contains the
   API's native ID. This makes the delete path provider-agnostic.

### 1.5 Deleted Record Treated as Drift Instead of Delete

**Problem:** When a user deletes a record from the database, the DiffEngine has no concept of
"this record should be removed from the provider." The provider still has the record, so it
appears as Drift. The user must manually select "delete" as a drift action during deployment,
which is unintuitive — the user already expressed intent to delete by removing the record.

**Design decision:** Implement a **pending-delete** model. When a user deletes a record, it is
soft-deleted (marked `pending_delete = true`) rather than hard-deleted. The DiffEngine sees
pending-delete records and generates explicit `Delete` diffs. After successful deployment, the
records are hard-deleted.

**Schema change (`scripts/db/v006/001_add_pending_delete.sql`):**

```sql
ALTER TABLE records ADD COLUMN pending_delete BOOLEAN NOT NULL DEFAULT FALSE;
```

**Backend changes:**

1. **`RecordRepository`** — change `deleteById()` to set `pending_delete = true` instead of
   `DELETE`. Add `hardDeleteById()` for post-deployment cleanup. Modify `listByZoneId()` to
   include pending-delete records with a flag.

   ```cpp
   // Soft delete
   void deleteById(int64_t iZoneId, int64_t iRecordId);
     // UPDATE records SET pending_delete = true WHERE id = $1 AND zone_id = $2

   // Hard delete (after successful push)
   void hardDeletePending(int64_t iZoneId);
     // DELETE FROM records WHERE zone_id = $1 AND pending_delete = true

   // List returns all records; caller filters by pending_delete
   ```

2. **`DiffEngine::preview()`** — separate records into two sets:
   - Active records (pending_delete = false) → compute Add/Update diffs as before
   - Pending-delete records → generate `Delete` diffs against live provider records

   For Delete diffs, match pending-delete records against live records by (name, type, value)
   to find the provider record ID. If a pending-delete record has no matching live record
   (already deleted from provider), skip it and hard-delete immediately.

   ```cpp
   // After computing normal diffs, process pending deletes:
   for (const auto& row : vPendingDeleteRows) {
     DnsRecord drDesired = /* expand row */;
     // Find matching live record
     for (const auto& drLive : vLive) {
       if (drLive.sName == drDesired.sName && drLive.sType == drDesired.sType
           && drLive.sValue == drDesired.sValue) {
         RecordDiff rd;
         rd.action = DiffAction::Delete;
         rd.sName = drLive.sName;
         rd.sType = drLive.sType;
         rd.sProviderValue = drLive.sValue;
         rd.sProviderRecordId = drLive.sProviderRecordId;
         rd.uTtl = drLive.uTtl;
         vDiffs.push_back(rd);
         break;
       }
     }
   }
   ```

3. **`DeploymentEngine::push()`** — implement the `Delete` case (currently a placeholder):

   ```cpp
   case common::DiffAction::Delete: {
     bool bDeleted = upProvider->deleteRecord(oZone->sName, diff.sProviderRecordId);
     if (!bDeleted) {
       throw common::ProviderError("PROVIDER_DELETE_FAILED",
                                   "Failed to delete record: " + diff.sName);
     }
     break;
   }
   ```

   After all diffs are applied successfully, call `_rrRepo.hardDeletePending(iZoneId)` to
   remove soft-deleted records from the database.

4. **`RecordRoutes`** — the `DELETE` endpoint already calls `deleteById()`. No change needed
   since `deleteById()` now performs a soft delete. The UI should show pending-delete records
   with visual distinction (strikethrough or faded).

**Frontend changes:**

- `ZoneDetailView.vue` — show pending-delete records with strikethrough text and a muted
  badge ("Pending Delete"). Add an "Undo Delete" action that clears the flag. Hide
  pending-delete records from the edit form.
- `DeploymentsView.vue` — Delete diffs render with red background and "DELETE" badge,
  matching the existing Add (green) and Update (amber) styling. No drift action selection
  needed for Delete diffs.

**Interaction with drift:** Records that exist on the provider but not in the database (and
were never managed) remain as Drift. Only records that were explicitly deleted by the user
(pending_delete = true) become Delete diffs. This preserves the existing drift resolution
workflow for genuinely unknown records.

## 2. UX Improvements

### 2.1 Record View Refresh Resetting Scroll Position and Row Count

**Problem:** Every record save in `ZoneDetailView.vue` calls `await fetchData()` which
re-fetches the full zone and records list, causing Vue to re-render the DataTable. This
resets scroll position to top and resets pagination to page 1.

**Fix:** Replace full refresh with optimistic local array update.

After `updateRecord()` returns the updated record JSON, find the record in the local
`records` array by ID and replace it in-place. After `createRecord()`, push the new record
to the array. Only call `fetchData()` on error as a fallback.

```typescript
async function handleSubmitRecord() {
  // ... validation ...
  try {
    if (editingRecordId.value !== null) {
      const updated = await recordApi.updateRecord(zoneId, editingRecordId.value, payload)
      const idx = records.value.findIndex(r => r.id === editingRecordId.value)
      if (idx !== -1) records.value[idx] = updated
    } else {
      const created = await recordApi.createRecord(zoneId, payload)
      records.value.push(created)
    }
    showRecordDialog.value = false
    notify.success(editingRecordId.value ? 'Record updated' : 'Record created')
  } catch (err) {
    // ... error handling ...
  }
}
```

Similarly, after delete confirmation succeeds, splice the record from the array by ID
(or mark it as pending-delete per §1.4):

```typescript
const idx = records.value.findIndex(r => r.id === deletedId)
if (idx !== -1) {
  records.value[idx].pending_delete = true  // Visual feedback
}
```

### 2.2 Sort Variables Alphabetically in Lookup

**Problem:** Variables are ordered by insertion ID (`ORDER BY id`) in the database. The
autocomplete panel and Variables management page show variables in arbitrary insertion order.

**Fix (backend):** Change all three `ORDER BY` clauses in `VariableRepository.cpp`:

```cpp
// listAll() — scope grouping, then alphabetical
"ORDER BY CASE WHEN scope = 'global' THEN 0 ELSE 1 END, name"

// listByScope() — alphabetical within scope
"ORDER BY name"

// listByZoneId() — globals first, then zone-scoped, alphabetical within each
"ORDER BY CASE WHEN scope = 'global' THEN 0 ELSE 1 END, name"
```

This puts global variables first (most commonly referenced), then zone-scoped, alphabetical
within each group.

## 3. New Features

### 3.1 Admin Password Reset CLI

**Problem:** If the only admin account is locked out (forgotten password, no other admin
users), there is no recovery path. The change-password API requires authentication, and the
password is Argon2id-hashed so direct SQL updates aren't practical.

**Solution:** Add a `--reset-password <username>` CLI flag to the main binary. This connects
to the database, prompts for a new password, hashes it with Argon2id, updates the user, and
sets `force_password_change = true`.

**Implementation (`src/main.cpp`):**

Add a new CLI branch alongside `--help`, `--version`, `--migrate`:

```cpp
if (arg == "--reset-password") {
  if (i + 1 >= argc) {
    std::cerr << "Usage: --reset-password <username>\n";
    std::quick_exit(EXIT_FAILURE);
  }
  std::string sUsername = argv[++i];
  try {
    auto cfgApp = dns::common::Config::load();
    dns::common::Logger::init(cfgApp.sLogLevel);

    dns::dal::ConnectionPool cpPool(cfgApp.sDbUrl, 1);
    dns::dal::UserRepository urRepo(cpPool);

    auto oUser = urRepo.findByUsername(sUsername);
    if (!oUser) {
      std::cerr << "User '" << sUsername << "' not found\n";
      std::quick_exit(EXIT_FAILURE);
    }

    // Read password from stdin (supports piping and interactive)
    std::string sNewPassword;
    if (isatty(STDIN_FILENO)) {
      std::cout << "New password for '" << sUsername << "': " << std::flush;
      // Disable echo for password input
      struct termios tOld, tNew;
      tcgetattr(STDIN_FILENO, &tOld);
      tNew = tOld;
      tNew.c_lflag &= ~ECHO;
      tcsetattr(STDIN_FILENO, TCSANOW, &tNew);
      std::getline(std::cin, sNewPassword);
      tcsetattr(STDIN_FILENO, TCSANOW, &tOld);
      std::cout << "\n";
    } else {
      std::getline(std::cin, sNewPassword);
    }

    if (sNewPassword.empty() || sNewPassword.size() < 8) {
      std::cerr << "Password must be at least 8 characters\n";
      std::quick_exit(EXIT_FAILURE);
    }

    std::string sHash = dns::security::CryptoService::hashPassword(sNewPassword);
    urRepo.updatePassword(oUser->iId, sHash);
    urRepo.setForcePasswordChange(oUser->iId, true);

    std::cout << "Password reset for '" << sUsername << "'. "
              << "User will be prompted to change password on next login.\n";
    std::cout.flush();
  } catch (const std::exception& ex) {
    std::cerr << "[fatal] password reset failed: " << ex.what() << std::endl;
    std::quick_exit(EXIT_FAILURE);
  }
  std::quick_exit(EXIT_SUCCESS);
}
```

**Usage:**

```bash
# Interactive (prompts for password with hidden input)
meridian-dns --reset-password admin

# Non-interactive (piped, for scripts/Docker)
echo "NewSecurePassword123" | meridian-dns --reset-password admin
```

**Includes:** `<unistd.h>` and `<termios.h>` for `isatty()` and echo suppression. Both are
POSIX and already available on the target platform (Linux).

**Help text update:** Add `--reset-password <user>` to `printUsage()`.

### 3.2 Bulk Record Editing

**Goal:** Allow users to edit multiple records without opening/closing a modal for each one,
and to perform bulk operations (delete, TTL change) across selected records.

**Approach: Inline table editing + multi-select bulk actions.**

**Frontend changes (`ZoneDetailView.vue`):**

1. **Inline cell editing** — Enable PrimeVue DataTable `editMode="cell"` for the Value, TTL,
   and Priority columns. Double-click a cell to edit inline. On blur or Enter, save the
   individual record via API.

2. **Row selection** — Add checkbox selection column. Selection enables a bulk action toolbar
   above the table with:
   - **Delete Selected** — soft-deletes all selected records (pending_delete per §1.4)
   - **Set TTL** — opens a small popover to set TTL for all selected records
   - **Set Type** — dropdown to change record type for all selected

3. **Batch API endpoint** — `PUT /api/v1/zones/{id}/records/batch`:
   ```json
   {
     "updates": [
       { "id": 1, "ttl": 3600 },
       { "id": 2, "ttl": 3600 },
       { "id": 5, "value_template": "10.0.0.2" }
     ],
     "deletes": [3, 7]
   }
   ```
   Returns the updated records array. Creates a single audit entry (see §3.2).

**Backend changes:**

1. **`RecordRepository::batchUpdate()`** — accepts vector of `{id, fields}` pairs. Executes
   in a single transaction. Returns updated records.

2. **`RecordRepository::batchSoftDelete()`** — sets `pending_delete = true` for a vector of
   record IDs in a single transaction.

3. **`RecordRoutes`** — new `PUT /api/v1/zones/{id}/records/batch` endpoint. Validates all
   records belong to the zone. Calls batch methods. Writes single audit entry.

**Interaction with §2.1:** Inline edits use the same optimistic update pattern — update the
local array cell value immediately, fire the API call, revert on error.

### 3.3 Record Change Audit Trail

**Problem:** Audit entries are only created during deployment push. Individual record CRUD
operations (create, update, delete) have no audit trail. Users cannot see who modified the
desired state before deployment.

**Design:**

Add `AuditRepository::insert()` calls to all record mutation endpoints in `RecordRoutes.cpp`:

| Operation | entity_type | entity_id | operation | old_value | new_value |
|-----------|------------|-----------|-----------|-----------|-----------|
| Create | `record` | record ID | `create` | null | `{name, type, value, ttl, ...}` |
| Update | `record` | record ID | `update` | `{old fields}` | `{new fields}` |
| Delete | `record` | record ID | `delete` | `{old fields}` | null |
| Batch update | `zone` | zone ID | `batch_update` | null | `{updates: [...], deletes: [...]}` |
| Batch delete | `zone` | zone ID | `batch_delete` | null | `{record_ids: [...]}` |

**Implementation:**

1. For **create**: after `_rrRepo.create()`, insert audit with new record data.
2. For **update**: fetch the existing record before update (`findById`), then insert audit
   with both old and new values.
3. For **delete**: fetch the existing record before soft-delete, insert audit with old values.
4. For **batch operations**: create a single audit entry with operation `batch_update` or
   `batch_delete`, with new_value containing the full changeset as JSON.

**Identity:** Use `rcCtx.sUsername` from the authenticated request context, already available
in all route handlers.

## 4. Decisions

### 4.1 Variable Syntax — No Change

Keep `{{NAME}}` syntax. The `$NAME` alternative has unacceptable collision risk in DNS record
values (SPF macros, TXT records, SRV targets). The double-brace delimiters provide unambiguous
start/end markers. The existing `{{` autocomplete trigger and browse button mitigate typing
overhead.
