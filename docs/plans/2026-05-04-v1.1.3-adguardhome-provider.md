# v1.1.3 — AdGuard Home Provider

## Overview

Adds AdGuard Home as a companion DNS provider, exposing its DNS Rewrite function through the
standard Meridian DNS provider interface. Records opted in to AdGuard Home are pushed as rewrites
alongside their authoritative provider (e.g. PowerDNS or Cloudflare), enabling split-horizon
resolution for local networks — for example, a record that is a Cloudflare Tunnel CNAME
externally can resolve to an internal IP via AdGuard Home on the LAN.

---

## Section 1: Provider Layer

**Type string:** `"adguardhome"` — registered in `ProviderFactory`.

**Authentication:** The encrypted token field stores JSON:
```json
{"username": "admin", "password": "secret"}
```
The provider parses this on construction and configures `httplib::Client` with
`Authorization: Basic <base64(user:password)>`.

**API endpoints used:**
- `GET /control/rewrite/list` — fetch all rewrites
- `POST /control/rewrite/add {"domain": "...", "answer": "..."}` — create
- `POST /control/rewrite/delete {"domain": "...", "answer": "..."}` — delete

AdGuard Home has no native PATCH for rewrites; `updateRecord()` is implemented as
delete-then-add.

**Record ID:** AdGuard Home has no native record ID. `sProviderRecordId` is encoded as
`{domain}|{answer}` (e.g. `www.example.com|192.168.1.100`). `deleteRecord()` splits on the
first `|` to reconstruct the delete payload.

**FQDN normalisation:** `listRecords()` adds trailing dots to domain and CNAME answer values.
`createRecord()` and `updateRecord()` strip trailing dots before sending to AdGuard Home.

**TTL:** AdGuard Home rewrites carry no TTL. The `uTtl` field is silently ignored when pushing.
`listRecords()` returns `uTtl = 0` for all rewrites.

**Connectivity check:** `GET /control/status` — parses the `running` boolean field and maps to
`HealthStatus::Ok`, `Degraded`, or `Unreachable`.

## Section 2: Record Metadata Model

AdGuard Home opt-in is stored in the existing `provider_meta` JSONB column on the `records`
table. No schema migration is required.

**Per-record metadata:**
```json
{
  "aghdns_enabled": true,
  "aghdns_answer": "192.168.1.100"
}
```

**`aghdns_enabled`** — bool, absent/false by default. The AdGuard Home provider skips any record
where this flag is absent or false.

**`aghdns_answer`** — string; the value sent to AdGuard Home as the rewrite answer. The record's
`sValue` is ignored by this provider. The answer format determines the rewrite type:
- Valid IPv4 → A rewrite
- Valid IPv6 → AAAA rewrite
- Valid hostname → CNAME rewrite

**Validation:** `aghdns_answer` must be a valid IPv4, IPv6, or RFC 1123 hostname. Enforced in
`AdGuardHomeProvider::createRecord` (throws `ProviderError`) and in `RequestValidator` at the
API boundary. The existing `isValidHostname()` method in `RequestValidator.cpp` is reused.

**UI restriction:** The toggle and answer field are only rendered when the zone is attached to at
least one AdGuard Home provider AND the record type is `A`, `AAAA`, or `CNAME`.

## Section 3: IProvider Interface Change

DiffEngine compares `(name, type, value)` tuples. For AdGuard Home the desired state is derived
from `aghdns_answer` (e.g. `A 192.168.1.100`), not the Meridian record's own type/value (e.g.
`CNAME tunnel.cfargotunnel.com`). Without a transformation step every opted-in record would
appear as permanent drift.

**New optional virtual method on `IProvider`:**
```cpp
virtual std::vector<DnsRecord> prepareDesiredRecords(
    const std::vector<DnsRecord>& vDesired) const {
  return vDesired;  // default: no-op
}
```

All five existing providers inherit the no-op default — zero impact on existing behaviour.

**`AdGuardHomeProvider` override:**
1. Filter: keep only records where `jProviderMeta["aghdns_enabled"] == true` and `sType` is
   `A`, `AAAA`, or `CNAME`
2. Transform: infer `sType` from `aghdns_answer` format; set `sValue = aghdns_answer`; preserve
   `sName`, `sProviderRecordId`, and `jProviderMeta`

**`DeploymentEngine` change** — one line added before each DiffEngine call:
```cpp
auto vProviderDesired = spProvider->prepareDesiredRecords(vDesired);
auto vChanges = _spDiffEngine->compute(vProviderDesired, vLive);
```

`createRecord` and `updateRecord` receive the transformed record. The provider uses
`jProviderMeta["aghdns_answer"]` directly (preserved through the transform).

## Section 4: UI Changes

**Record form (`ZoneDetailView.vue`):** When the zone is attached to at least one AdGuard Home
provider, a collapsible "AdGuard Home" section appears below the standard fields, styled
consistently with the existing Cloudflare proxy section. Rendered only for `A`, `AAAA`, and
`CNAME` record types.

Contents:
- **Toggle:** "Enable AdGuard Rewrite" (`aghdns_enabled`) — PrimeVue `ToggleSwitch`
- **Text field:** "Rewrite Answer" (`aghdns_answer`) — visible only when toggle is on;
  placeholder: `192.168.1.1 / ::1 / host.local`; validates on blur with inline error for invalid
  format

**Provider form (`ProvidersView.vue`):** `"adguardhome"` added to the type dropdown. Selecting
it shows:
- **Endpoint** — URL of AdGuard Home instance (e.g. `http://192.168.1.1:3000`)
- **Username** and **Password** fields — combined client-side into JSON token format before
  submission; password field has show/hide toggle

**Zone detail record list:** Records with `aghdns_enabled: true` display a small "AdGuard" badge,
consistent with the existing Cloudflare proxy badge pattern.

## Section 5: Testing

**Unit tests** (`tests/unit/test_adguardhome_provider.cpp`) — no live instance required (~15
tests):
- `prepareDesiredRecords` filtering: excludes records without `aghdns_enabled`; excludes
  unsupported types (MX, TXT) even if `aghdns_enabled` is set
- `prepareDesiredRecords` transformation: sType and sValue derived correctly from `aghdns_answer`
  for IPv4, IPv6, and hostname; jProviderMeta and sName preserved
- FQDN normalisation: trailing dots added on `listRecords()` output, stripped before API calls
- `sProviderRecordId` encoding/decoding: `domain|answer` round-trips correctly
- JSON parsing of `listRecords()` response: well-formed and malformed payloads handled
- `updateRecord()` issues delete-then-add
- `testConnectivity()` status mapping: `running: true` → Ok, `running: false` → Degraded,
  connection failure → Unreachable

**Integration tests** (`tests/integration/test_adguardhome_provider.cpp`) — gated behind
`DNS_AGH_URL` environment variable, skipped when absent (~3 tests):
- Full CRUD cycle: create → list → update → list → delete → list
- Type inference round-trip: push IPv4 answer, verify `listRecords()` returns `A` type

**Conformance test:** Add `"adguardhome"` to the existing provider conformance suite to validate
IProvider contract compliance.

**Estimated new tests:** ~18 unit + 3 integration = 21 total.
